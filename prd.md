## 1. 产品目标

五个正交的核心目标，共同定义产品的战略方向：

### 目标一：消除 AI 对话产出的"数字黑洞"
**问题**：AI Agent 在对话中生成的内容（分析报告、文档、代码、总结等）困在对话历史中，需要用户手动复制→粘贴→格式化才能保存和分享。  
**衡量**：用户从"Agent 输出完成"到"内容可分享给他人"的步骤数。当前 `≥5` 步，目标降至 `≤2` 步。

### 目标二：建立 Agent → 人的主动交付闭环
**问题**：AI 产出了但用户不知道。内容生成后缺乏主动通知机制，用户需要主动"上线查看"。  
**衡量**：内容发布后 30 分钟内的小程序通知送达率 ≥99%；通知点击率 ≥30%（对标微信订阅消息行业均值）。

### 目标三：成为 AI Agent 生态的标准内容发布层
**问题**：多 Agent（Claude/ChatGPT/Gemini/Cursor 等）、多模型场景下，内容碎片化散落在不同对话中，缺乏统一的跨 Agent 内容仓库。  
**衡量**：上线 6 个月内对接 ≥3 个主流 Agent 框架（OpenClaw / WorkBuddy / Claude Code / Cursor 等）；北极星指标为**每周通过 AI Agent 发布的可分享内容数**。

### 目标四：移动端优先的消费体验
**问题**：用户和内容接收方大量在手机端浏览内容，但 AI 生成的 Markdown 未经适配直接在手机端阅读效果差，排版错乱、代码溢出、表格不可读。  
**衡量**：移动端（<768px 宽度）分享链接的阅读满意度 ≥4.0/5.0；移动端页面加载时间 ≤2 秒；手机浏览器打开所有分享链接即获得最佳阅读体验（零手动缩放操作）。

### 目标五：从内容堆砌到知识库沉淀
**问题**：AI Agent 持续输出内容后，用户在个人空间中面临大量碎片化内容堆积，缺乏结构化组织和系统性知识管理的能力，内容"存了用不上"。  
**衡量**：上线 3 个月内，用户创建的 **知识库分组数 / 人均内容分类率** ≥60%（即 60% 以上的已发布内容被归入某个知识分组）；知识库分享链接在整体分享链接中占比 ≥20%。

## 2. 用户故事

### 2.1 AI Agent 用户 Allen — 一键发布，手机直达

> **作为**使用 AI Agent（如 OpenClaw、WorkBuddy）的用户，  
> **我希望**在 Agent 对话中让 AI 将输出产物（Markdown）一键上传到系统，返回一个可直接手机预览访问的地址，  
> **以便**我直接把链接发给他人，对方在手机上打开即可阅读内容，无需任何额外操作。

**验收标准**：
- **Given** Allen 在 AI Agent 对话中调用发布 Skill，输入为 Markdown 文本；  
  **When** 发布成功；  
  **Then** 系统返回一个短链接，该链接在手机浏览器打开时自动以移动端优化布局渲染（字体 ≥16px、行高 ≥1.6、代码块可横向滚动、图片自适应屏幕宽度），无需手动缩放。
- **Given** Allen 将链接分享给客户；  
  **When** 客户在手机微信/浏览器中点击链接；  
  **Then** 页面在 2 秒内完成渲染，内容全文可读，无布局错乱。
- **Given** 该链接在 PC 端打开；  
  **When** 浏览器宽度 ≥768px；  
  **Then** 渲染引擎自动切换为多栏宽版布局，充分利用屏幕空间。

### 2.2 独立开发者 Bob — 技术文档一站展示

> **作为**一名独立开发者，  
> **我希望**AI 为我生成的 SDK 文档直接推送到我的个人技术站点（含代码高亮 + 自动目录），并通过我自己的域名访问，  
> **以便**我的用户能直接阅读专业文档，我不需要自己搭文档站点。

**验收标准**：
- **Given** Agent 推送的 Markdown 内容包含代码块；  
  **When** 渲染引擎处理完毕；  
  **Then** 自动应用语法高亮（支持 JavaScript/Python/Go/Java/Rust 等 ≥10 种语言）。
- **Given** Markdown 内容包含 h1-h6 标题层级；  
  **When** 渲染输出 HTML；  
  **Then** 自动生成可跳转目录（TOC），移动端目录折叠为汉堡菜单。
- **Given** 用户想在手机端查看文档；  
  **When** 在手机浏览器打开分享链接；  
  **Then** 代码块支持横向滑动查看，目录可折叠，正文无溢出。
- **Given** Bob 完成自定义域名验证；  
  **When** 绑定成功；  
  **Then** 所有分享链接可通过自有域名访问（v1.5 功能）。

### 2.3 研究生 Cathy — 文献积累与知识库构建

> **作为**一名在读研究生，  
> **我希望**AI 总结的论文推送到个人空间后可以添加标签、分类归档到知识库分组，并把单篇文献总结或整个文献知识库分享给导师，  
> **以便**我逐步建立起自己的结构化文献知识库，并灵活控制分享范围。

**验收标准**：
- **Given** Agent 推送时传入标签参数（如 `tags: ["NLP", "Transformer", "2026"]`）；  
  **When** 内容发布成功；  
  **Then** 标签在个人空间和内容页中展示，支持点击筛选。
- **Given** Cathy 在个人空间创建知识库分组（如"2026 顶会论文"）；  
  **When** 将多篇文献总结归类到该分组；  
  **Then** 分组内按时间倒序展示所有文献，支持分组级链接分享。
- **Given** 用户通过手机访问知识库分组链接；  
  **When** 页面加载；  
  **Then** 以移动端优化列表展示分组内所有内容标题 + 摘要，点击进入单篇详情的渲染布局同样为移动端优先。
- **Given** Cathy 分享单篇文献给导师；  
  **When** 导师在手机上打开链接；  
  **Then** 显示内容标题、发布时间、标签、完整渲染内容，移动端阅读体验流畅。

### 2.4 自媒体系列运营者 Diana — 内容排版的"预览验证"

> **作为**一名自由撰稿人，  
> **我希望**AI 生成公众号初稿后一键推到个人空间预览排版效果，再把链接发给合作方确认，确认后再发到公众号，  
> **以便**减少反复修改排版的沟通成本。

**验收标准**：
- **Given** Agent 推送含标题、子标题、列表、引用、图片链接的 Markdown；  
  **When** 渲染引擎处理完毕；  
  **Then** 在移动端（<768px）以单列布局自适应显示，PC 端多栏展示。
- **Given** 分享链接在微信内打开；  
  **When** 页面加载；  
  **Then** 自动适配微信内置浏览器视口宽度，移动端优先布局。
- **Given** Diana 将内容设为"草稿"状态；  
  **When** 确认发布前；  
  **Then** 内容仅自己可见，分享页面返回"内容暂未发布"。
- **Given** 内容需要更新；  
  **When** Diana 发布新版本；  
  **Then** 旧版本在版本历史中可追溯（v2 功能）。

### 2.5 AI Agent 运营者 Evan — 团队知识仓库

> **作为**一家 SaaS 公司的 AI 产品运营，  
> **我希望**公司内部多个 Agent（客服总结 Agent、周报生成 Agent、竞品分析 Agent）统一推送到团队空间，  
> **以便**团队所有人能在一个地方查看所有 AI 生成的输出，形成团队知识库。

**验收标准**：
- **Given** Evan 创建团队空间并邀请成员（v2 功能）；  
  **When** 成员接受邀请；  
  **Then** 空间内所有内容对所有成员可见。
- **Given** 多个 Agent 在同一空间推送内容；  
  **When** 团队成员查看空间；  
  **Then** 支持按 Agent 来源/时间/标签筛选。
- **Given** 成员在手机上访问团队空间链接；  
  **When** 页面加载；  
  **Then** 移动端布局展示内容列表，点击进入详情同样为移动端优先渲染。

### 2.6 日常查看者 Frank — 今日内容聚合视图

> **作为**一名高频使用 AI Agent 的日常用户，  
> **我希望**进入个人空间后能直接看到今天（含定时任务）生成的所有内容，并按时间维度（今日/本周/本月）来筛选查看，  
> **以便**我快速掌握当天 AI 为我产出了什么，不必在一堆旧内容中翻找。

**验收标准**：
- **Given** Frank 登录个人空间；  
  **When** 进入"今日视图"；  
  **Then** 默认显示当天（00:00-23:59）发布的所有内容，按时间倒序排列。
- **Given** Frank 希望查看更长时间范围；  
  **When** 点击"本周"或"本月"标签；  
  **Then** 内容列表刷新为对应时间范围内的内容。
- **Given** AI Agent 的定时任务（如日报生成）在凌晨自动完成并推送；  
  **When** 当天早上 Frank 打开个人空间；  
  **Then** 日报内容出现在今日视图中，且标记来源为"scheduled（定时任务）"。
- **Given** Frank 在手机上访问今日视图；  
  **When** 页面加载；  
  **Then** 移动端优化列表展示，每条内容可点击进入详情，详情页同样为移动端优先渲染。

### 2.7 知识库构建者 Grace — 结构化知识管理

> **作为一名**深度知识工作者，  
> **我希望**将 AI 持续产出的内容按照主题/项目整理为结构化的知识库（创建分组、建立目录层级），并能将整个知识库一键分享给团队或客户，  
> **以便**我不再面对"内容堆砌"的混乱局面，而是拥有可管理、可导航、可分享的结构化知识资产。

**验收标准**：
- **Given** Grace 在个人空间创建知识库分组（如"Q2 市场分析"）；  
  **When** 创建成功；  
  **Then** 分组出现在个人空间的"知识库"目录树中。
- **Given** Grace 选择多篇已发布内容；  
  **When** 将其归类到某个分组；  
  **Then** 内容的 `group_id` 更新为该分组 ID，在目录树中对应分组下可查看到该内容。
- **Given** Grace 希望分享整个知识库；  
  **When** 生成知识库共享链接；  
  **Then** 接收方打开链接后看到该分组下的所有内容列表（标题 + 摘要 + 发布时间），点击任意标题进入单篇详情。
- **Given** 知识库分组需要子分类（v1 支持 1 级嵌套）；  
  **When** Grace 在已有分组下创建子分组；  
  **Then** 目录树以层级结构展示父子关系（最多支持 2 级）。
- **Given** 接收方在手机上打开知识库共享链接；  
  **When** 页面加载；  
  **Then** 移动端优先布局，分组以可折叠的目录树形式展示。

## 3. 用户研究洞察（来源：瑞思/Reese）

### 3.1 七大核心痛点（按严重程度排序）

| 等级 | 痛点 | 说明 | 频率 |
|------|------|------|------|
| ★★★★★ | AI 对话产出 = "数字黑洞" | 输出困在对话历史，需要频繁复制→粘贴→格式化 | 每次 AI 对话 |
| ★★★★★ | AI 无法主动通知 | "产出了但用户不知道" | 每次内容生成 |
| ★★★★★ | AI 内容缺乏知识化管理 | 内容发布后缺乏结构化整理能力，堆积成"内容垃圾场"，无法按主题/项目检索和管理 | 持续积累后 |
| ★★★★☆ | 内容分享链路断裂 | "写好了但没法体面地发出去" | 需要分享的场合 |
| ★★★★☆ | 多 Agent/多模型内容碎片化 | 内容散落在不同对话中 | 使用多个 Agent 的用户 |
| ★★★☆☆ | Markdown → 可分享页面转换门槛 | 需要额外工具渲染美化 | 每次需要展示时 |
| ★★★☆☆ | 内容版本迭代缺乏追踪 | 更新后旧版消失，无法回溯 | 内容反复修改时 |

### 3.2 用户旅程关键断裂带

```
对话完成 [✓] → 复制内容 [✓] → 整理格式 [!] → 美化排版 [✗] → 生成链接 [✗] → 分享给他人 [✗]
                              ↑                 ↑                      ↑                    ↑
                         频率最高痛点      门槛最高的环节        最缺失的能力          最终目的被阻断
```

**核心洞察**：断裂集中在③整理→④美化→⑤生成链接→⑥分享这个链条上。我们的产品定位就是消除③~⑥之间的所有摩擦。

### 3.3 用户心理模型

> **"AI 对话的延伸——对话结束了，内容留下了。"**

- **个人空间 = 作品展示墙 + 知识库**，不是后台文件管理器 —— UI 应像画廊式展示，而非表格列表；新增"今日视图"提供时间维度的聚合入口
- **小程序 = 信使**，不是主要阅读工具 —— 通知为主、轻量预览为辅，详细阅读引导到浏览器
- **知识库 = 精品展柜**，不是文件夹 —— 用户将精选内容归入知识库分组后，展示时应像"主题策展"，而非目录树，整体分享时接收方体验应像阅读一本小册子
- **不是"又一个博客平台"** —— 用户不会在这里撰写内容，内容来自 AI Agent

### 3.4 关键设计原则

| # | 原则 | 含义 |
|---|------|------|
| 1 | **Agent 内触发是第一入口** | Skill 调用是发布内容的唯一入口，Web 端是管理/查看的后台 |
| 2 | **零配置渲染** | 用户不需要选择主题、配置样式，开箱即出精美页面 |
| 3 | **分享即用，无需注册** | 分享链接的接收方不需要注册账号即可阅读内容 |
| 4 | **通知是核心功能** | 内容发布成功时必须主动通知用户，这不是增值功能而是基线功能 |
| 5 | **个人空间 = 作品集** | 展示优先于管理，默认视图为画廊/卡片式布局 |
| 6 | **AI 输出格式即最终格式** | 不要要求用户在发布后二次编辑格式，Markdown 入→HTML 出，一步到位 |
| 7 | **移动端优先，而非适配** | 所有分享链接默认以移动端布局渲染，PC 端自动升级为多栏宽版，而非反过来 |
| 8 | **内容积累 → 知识结构化** | 通过知识库分组、今日视图等能力，帮助用户从被动接收内容升级为主动构建知识体系 |

### 3.5 需求优先级矩阵

| 优先级 | 说明 | 核心需求 |
|--------|------|----------|
| **P0** | 缺了就无法发布 | Agent 一键推送、Markdown→HTML 渲染、分享链接生成（移动端优先）、微信小程序通知 |
| **P1** | 核心体验必须 | 个人空间管理（列表/标签/搜索）、**今日视图**（今日/本周/本月筛选）、**知识库分组/目录**、**定时任务内容接入**、**Mobile-First 预览页面**、小程序内预览、浏览统计、**访问密码保护**、草稿/发布状态 |
| **P2** | 差异化竞争力 | 版本历史、自定义主题/CSS、团队空间/协作、内容导出、**知识库整体分享**、**嵌套分组**、自定义域名 |
| **P3** | 生态扩展 | Agent 开发者 API、评论/反馈系统、内容发现/广场 |

## 4. 竞品对比（来源：竞析/Compa）

### 4.1 四类竞品全景

| 类别 | 代表产品 | 我们的差异点 |
|------|---------|-------------|
| **Markdown 渲染/展示** | GitHub Pages, GitBook, VitePress, Docusaurus, docsify, MkDocs, WordPress | 零技术门槛，无需 Git/构建/部署，AI Agent 一键发布 |
| **内容发布与分享** | 语雀, Notion, 飞书文档, Medium, 知乎专栏, 掘金, Substack | 纯 Markdown 输入，不设复杂编辑器，AI Agent 原生对接 |
| **AI Agent × 内容结合** | Claude Artifacts, ChatGPT Canvas, WordPress MCP, 语雀 Skill, Notion API | 内容永久保存（非对话生命周期），跨 Agent 统一仓库，微信通知闭环，**移动端优先渲染**，**知识库分组整理** |
| **小程序通知** | 微信一次性/长期订阅消息 | 内容更新直达用户微信，形成"发布→通知→阅读"闭环 |

## 8. Non-goals（明确不做什么）

以下内容**不在 v1（MVP+）范围内**，也不在 v2 的默认计划中，只有明确决策后才纳入：

### 8.1 不是内容创作平台
- ❌ 不提供在线富文本/Markdown 编辑器（内容全部来自 AI Agent）
- ❌ 不提供图片上传和图床功能
- ❌ 不支持在 Web 端直接撰写/编辑内容

### 8.2 不是社交/社区平台
- ❌ 不提供内容发现/推荐/广场功能（v1）
- ❌ 不支持评论、点赞、关注（v1）
- ❌ 不做内容排名、热度算法
- ❌ 不设用户等级、积分体系

### 8.3 不是通用 CMS
- ❌ 不提供页面模板系统（v1）
- ❌ 不支持多级页面导航、自定义菜单
- ❌ 不提供 SEO 优化配置（v1）
- ❌ 不设插件/扩展体系

### 8.4 不是协作平台
- ❌ 不支持多人实时协作编辑（v1）
- ❌ 不提供评论/批注/修订功能
- ❌ 不设工作流审批机制

### 8.5 不是商业化平台
- ❌ 不设付费墙、打赏、订阅变现功能（v1）
- ❌ 不做广告投放系统
- ❌ 不提供内容版权管理（DRM）

### 8.6 不是定时任务调度器（新增）
- ❌ 不管理定时任务的创建、编辑、调度、执行
- ❌ 不提供定时任务配置 UI
- ❌ 只接收 Agent 已完成定时任务的内容推送（通过 Webhook）

### 8.7 技术架构非目标
- ❌ 不做 PWA 离线访问（v1）
- ❌ 不提供原生 App（iOS/Android）
- ❌ 不做服务端渲染 SSR（初期用 CSR 即可）

## 10. 待确认问题

以下问题是截至目前尚未形成明确结论的开放决策点，需要团队讨论或进一步验证后确定：

### 10.1 产品策略

| # | 问题 | 背景 | 建议决策人 |
|---|------|------|-----------|
| Q1 | **产品名称是什么？** | PRD 中暂用"Markdown 内容展示与分享平台"，需要一个正式产品名称 | 产品总监 |
| Q2 | **免费版的核心限制是什么？** | 存储空间？内容数量？分享链接访问量？知识库分组数量？需要明确定义免费层和使用限制 | 产品总监 + 商业化 |
| Q3 | **D1 留存 35-45% 的目标是否可接受？** | 作为工具型产品，是否需要在 MVP 中就加入"次日回访钩子"（如每日 AI 总结推送、内容定时发布）来提高留存？**新增的今日视图和定时任务有望提升留存** | 产品总监 + 数据分析 |
| Q4 | **是否在 MVP 阶段就支持多 Agent 接入？** | 竞品分析强烈建议抢占 Agent 生态先发优势，但多 Agent 对接会增加 MVP 工作量。**OpenClaw 和 WorkBuddy 已明确为初期对接目标** | 产品总监 |

### 10.2 技术架构

| # | 问题 | 背景 | 建议决策人 |
|---|------|------|-----------|
| Q5 | **存储方案选型？** | 需要同时存 Markdown 原始内容和渲染后的 HTML。SQLite（低成本）vs PostgreSQL（可扩展）vs 对象存储+DB 混合方案？ | 技术负责人 |
| Q6 | **渲染引擎在服务端还是客户端？** | 服务端渲染（效果确定、SEO 好但服务器开销大）vs 客户端渲染（轻量后端但依赖 JS）vs 混合方案？**Mobile-First CSS 需在服务端渲染中嵌入** | 技术负责人 |
| Q7 | **微信小程序审核策略？** | 一次性订阅消息的模板审核可能被拒。是否预备案多个模板？是否准备备选方案（如公众号模板消息）？ | 技术负责人 + 法务 |
| Q8 | **Skill 优先对接哪个 Agent 框架？** | OpenClaw vs WorkBuddy vs Claude Code vs Cursor vs 自研 Skill 标准？**用户场景已明确 OpenClaw 和 WorkBuddy 为第一梯队** | 技术负责人 |

### 10.3 用户研究

| # | 问题 | 背景 | 建议决策人 |
|---|------|------|-----------|
| Q9 | **是否需要用户注册流程？还是用微信一键登录？** | 用户研究指出"分享即用无需注册"是设计原则，但创作者本人是否需要注册？使用微信 OAuth 还是邮箱注册？ | 产品总监 + 设计 |
| Q10 | **目标用户的第一批获客渠道？** | AI Agent 用户社区（V2EX/即刻/推特/X）？AI 工具整合营销？微信生态裂变？**OpenClaw 和 WorkBuddy 的用户群是核心种子池** | 产品总监 + 市场 |

### 10.4 法律与合规

| # | 问题 | 背景 | 建议决策人 |
|---|------|------|-----------|
| Q11 | **AI 生成内容的责任归属？** | 平台对 AI 生成内容的版权、真实性、合规性承担多大责任？是否需要用户协议中明确免责条款？ | 法务 |
| Q12 | **内容审核机制是否需要在 MVP 中就包含？** | 如果用户发布违规内容分享给他人，平台是否需要承担审核责任？MVP 阶段是否先做好举报和下线机制？ | 法务 + 产品总监 |

### 10.5 新增待确认问题

| # | 问题 | 背景 | 建议决策人 |
|---|------|------|-----------|
| Q13（新增） | **知识库分组是否支持嵌套？** | 用户场景2需求涉及结构化知识库。v1 建议支持 1 级分组即可（简单目录树），后续支持多级嵌套。**需确认 Phase 1 中知识库分组 MVP 的范围是否包含嵌套** | 产品总监 |
| Q14（新增） | **定时任务对接方案？** | 用户场景3提到 AI Agent 定时任务（如日报）自动推送。是否需要 Agent 框架（OpenClaw / WorkBuddy）侧提供 Webhook 回调？是否需要定义统一的标准 Webhook 协议？**定时任务的创建/调度在 Agent 侧完成，我们只接收已完成的内容** | 技术负责人 |
| Q15（新增） | **Mobile-First 渲染的断点策略？** | 移动端优先布局需要明确响应式断点方案。建议采用 `<768px` 为移动端、`≥768px` 为 PC 端的单断点方案，或更精细的多断点（手机 <480px、平板 480-768px、PC >768px）方案？ | 技术负责人 + 设计 |
| Q16（新增） | **今日视图的时间范围和默认行为？** | "今日"按自然日（00:00-23:59）还是最近 24 小时？是否支持自定义时间范围？默认进入个人空间时是否默认展示"今日视图"而非"全部内容"？ | 产品总监 |
